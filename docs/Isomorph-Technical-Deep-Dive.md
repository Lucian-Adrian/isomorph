# Isomorph DSL: A Technical Deep Dive

## Introduction

**Isomorph** is a custom Domain Specific Language (DSL) specifically designed for defining, analyzing, and rendering complex software architecture diagrams. It serves as a single source of truth for representing the static and dynamic structures of an application, moving away from drag-and-drop tools and replacing them with a strict, readable, versionable text format.

### Why Isomorph?
Isomorph bridges the gap between codebase and documentation. By utilizing a text-based format (`.isx`), development teams can:
- Version Control diagrams using standard Git workflows.
- Auto-generate models such as Class, Use Case, Sequence, Component, Flow, and Deployment diagrams.
- Semantically analyze representations to prevent modeling flaws (e.g., referential integrity breaks, circular inheritances).

---

## Theoretical Foundations: Formal Languages & Automata

At its core, Isomorph is founded on **Formal Language Theory**. The DSL is formally described as a **Context-Free Grammar (CFG)**, placing it within the Type-2 category of the Chomsky hierarchy. The language specifications are expressed using Extended Backus-Naur Form (EBNF) constructs and are documented using an ANTLR4 grammar (`Isomorph.g4`), which acts as the mathematical spec of the language's determinism and unambiguity.

The compiler pipeline is designed as a classic multi-stage frontend:
1. **Lexical Analysis** (Finite Automata)
2. **Syntactic Analysis** (CFG & Pushdown Automata concepts)
3. **Semantic Analysis** (Context-Sensitive Symbol Tracking)
4. **Graph Generation & Rendering** (IOM Translation)

---

## Phase 1: Lexical Analysis (The DFA)

The first phase of the Isomorph engine involves the **Lexer** (`src/parser/lexer.ts`). The lexer acts as a **Deterministic Finite Automaton (DFA)** that consumes the raw character stream (the source `.isx` code) and translates it into a discrete sequence of structural *Tokens*.

### The Maximal Munch Principle
The Isomorph DFA naturally applies the "maximal munch" principle. At any specific branch, the automaton attempts to consume the longest possible sequence of characters that matches a valid token.
Consider the relation operators in Isomorph:
- `--` (Association)
- `-->` (Directed Association)
- `--|>` (Inheritance)

If the engine encounters `--|>`, the DFA transitions through states for `-`, `-`, `|`, and `>` rather than terminating early at the `--` state.

### Ambiguity Resolution in Automata
One of the most complex lexical challenges involves handling **Stereotypes** (`<<Entity>>`) versus **Nested Generics** (`Map<String, List<Book>>`).
If stereotype closers were strictly lexed as `>>` (a single monolithic token), the lexer would crash on a nested generic where two closure brackets align. To elegantly bypass this lexical ambiguity without requiring Context-Sensitive parsing, Isomorph's lexer consumes two individual `GT` (`>`) tokens and resolves their semantic linkage further down the pipeline in the parser.

---

## Phase 2: Syntactic Analysis (Parser & AST)

Once the DFA has produced a token stream, it flows into the **Syntactic Analyzer** (`src/parser/parser.ts`). The parser ensures that the stream adheres to the rules defined by the CFG.

### LL(1) Recursive Descent
While the formal grammar exists in ANTLR4 format, the engineering team opted for a **Hand-Written Recursive Descent Parser**. The grammar is fundamentally **LL(1)** (Left-to-right processing, Leftmost derivation, 1 token lookahead), though the custom parser allows for tactical **LL(k)** lookaheads in complex local branches.

Why hand-written instead of generated?
1. **Rich Error Recovery:** By managing the stack frames manually, the parser can jump to synchronization points (like a semicolon `;` or closing brace `}`) when an error occurs, allowing the parser to catch *multiple* syntax errors in a single pass instead of halting at the first failure.
2. **Highly Contextual Diagnostics:** The parser understands the *intent* of the code. If you type `List<Book`, it doesn't just say `Unexpected EOF`; it natively reports `Expected '>' to close generic type argument list.`

### The Abstract Syntax Tree (AST)
The theoretical derivation generated by the parser is reduced into an **Abstract Syntax Tree (AST)**. This is an acyclic tree ($O(N)$ operations for traversal) representing the hierarchical structure of the document, strictly mapped directly onto TypeScript data interfaces.

---

## Phase 3: Static Semantics (Logic & Meaning)

An AST guarantees grammatical correctness, but not logical sanity. The phrase "Colorless green interfaces sleep furiously" is syntactically correct in CFGs but semantically impossible. The **Semantic Analyzer** (`src/semantics/analyzer.ts`) handles meaning.

Because these rules are *context-sensitive*, they exceed the mathematical capabilities of Context-Free Grammars (Type-2 languages) and require Turing-complete traversal passes.

### The Multi-Pass Validation Architecture
The analyzer implements two primary depth-first passes over the AST:
1. **The Declaration Pass (Symbol Tables):** It walks the tree to discover universally named entities (Classes, Actors, Components) and populates a high-performance Hash Map, establishing the global operational environment.
2. **The Resolution & Linking Pass:** It traverses the bodies and relations, rigorously resolving identifier strings (e.g. `extends BaseEntity`) against the generated Symbol Table to create hard memory references.

### Strict Semantic Constraints (SS)
The engine enforces comprehensive theoretical invariants via the **Isomorph Object Model (IOM)** graph:
* **SS-3: Referential Integrity:** A relation `A --> B` aborts compilation if node `B` does not exist in the symbol environment.
* **SS-6: Acyclic Directed Graphs (DAG):** Detects forbidden cyclical inheritance chains (`A -> B -> A`) by walking the resolution graphs and employing cycle-detection bounds.
* **SS-9: Domain Bounding:** Validates that polymorphic types adhere to diagram constraints (e.g., rejecting an `actor` being instantiated inside a heavily-structured `class` diagram domain).

## The Syntax and Grammar

Isomorph uses a curly-brace syntax familiar to developers from languages like Java, C#, or TypeScript.

### Structure of an Isomorph File

```isomorph
diagram MySystem : class {
  package inner_domain {
    // Entities go here
  }

  // Relations go here
}
```

### Supported Diagram Kinds
The environment dynamically shifts its allowed CFG sub-grammar depending on the root classification: `class`, `usecase`, `sequence`, `component`, `flow`, `deployment`.

### Entity Definitions
**Classes and Interfaces** support C-style inheritance and member declarations:

```isomorph
abstract class Book <<Entity>> extends BaseEntity implements Borrowable {
  + title: string
  + isbn: string
  - stock: int = 0
  + checkOut(user: string): bool
}

enum BookStatus {
  AVAILABLE
  CHECKED_OUT
}
```

*Modifiers*: `public (+)`, `private (-)`, `protected (#)`, `package (~)`, `abstract`, `static`, `final`.
*Typing*: Supports primitives (`int`, `string`, `bool`) and generics (`List<Book>`).

### Relations
Relations use simple text-based ASCII arrows. The direction and line style dictate semantic meaning to the LL(1) parser.

- Association: `--`
- Directed Association: `-->`
- Inheritance / Generalization: `--|>`
- Realization: `..|>`
- Aggregation: `--o`
- Composition: `--*`
- Dependency: `..>`

You can attach metadata attributes to relations:
```isomorph
Library --* Book [label="manages", toMult="0..*"]
```

## Example: The Library Management System

Below is how a structurally sound configuration looks when processed entirely through the Lexer $\rightarrow$ AST $\rightarrow$ IOM pipeline.

```isomorph
diagram LibrarySystem : class {
  package domain {
    abstract class BaseEntity {
      # id: int
    }

    class Library {
      + name: string
      - books: List<Book>
      - members: List<Member>
    }

    class Member {
      + name: string
      + borrow(book: Book): bool
    }

    interface Borrowable {
      + isAvailable(): bool
    }

    abstract class Book extends BaseEntity implements Borrowable {
      + title: string
      - stock: int
    }
  }

  // Compositions and Associations
  Library --* Book   [label="manages", toMult="0..*"]
  Library --* Member [label="has", toMult="0..*"]
  Member  --> Book   [label="borrows", toMult="0..*"]
}
```

## Conclusion

By bridging deep formal language concepts with robust engineering practices, Isomorph acts as far more than a simple drawing tool. It enforces mathematical invariants (DAG constraints, referential integrity) at the architecture design layer. By generating a heavily validated intermediate representation (the IOM), compiling to architecture diagrams becomes fundamentally deterministic, reliable, and firmly integrated within modern typed-system CI/CD lifecycles.
