The Status Quo

The industry standard
is fundamentally broken.

Engineering teams are paralyzed by a decades-old dichotomy in systems design.

Option A

Textual Tools.

PlantUML, Mermaid.
Perfect for Git diffs. Terrible for layout. One line of changed code can scramble a carefully orchestrated architecture diagram entirely.

Visual Editors.

draw.io, Excalidraw, Figma.
Beautiful, stable layouts. But stored in opaque binary or dense XML formats. Completely unreadable in pull requests.

Operators

Structural vs behavioral intent.

Commands and operators in the DSL describe both structure (types/relationships) and behavior (flows/events). The editor keeps both in sync.

The Solution

A domain-specific language
where layout is source data.

parse(serialise(M)) = M
The Absolute Truth

The only invariant
that matters.

Bidirectional Architecture

One Tool.
One Truth.

Edit the text, the canvas updates.

Drag an entity on the canvas, the code updates.

The model stays completely canonical.

Layout lives inline in the text; every drag is a deterministic edit.

Syntax Design

Coverage

Nine complete UML
diagram types.

Class, usecase, sequence, component, deployment, state, activity, collaboration, and flow. Unified under one parser.

Phase 01

Lex.

66

Unique token kinds. Longest-match-first scanning. Zero external dependencies.

Phase 02

Parse.

55

Hand-written LL(1) production rules yielding a strictly typed 22-node AST.

Phase 03

Analyse.

14

Two-pass semantic static rules evaluating constraints, referential integrity, and cyclic inheritance.

Phase 04

Render.

09

Pixel-perfect SVG renderers converting abstract syntax trees into interactive visual geometry.

ELK Engine

Automatic Layout.

Entities without @annotations are mathematically placed using the Eclipse Layout Kernel. It guarantees stability across re-parses.
